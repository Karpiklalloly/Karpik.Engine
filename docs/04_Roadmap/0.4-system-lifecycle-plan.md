# 0.4 System Lifecycle Plan

## Summary

Goal: move user modules away from direct DragonECS lifecycle APIs and make KarpikEngine own the system pipeline.

DragonECS remains the internal ECS backend for 0.4. Aspects, `IEcsComponent`, `EcsPool<T>` inside aspects, and existing Dragon internals can stay for now. The public direction is different: user systems implement Karpik `ISystem*` contracts, receive Karpik world facades, and let the engine scheduler decide which simulation systems can run in parallel.

The frame pipeline:

1. `Init` once when pipeline is built.
2. `Begin` every frame, sequential/main-thread.
3. `FixedRun` for fixed simulation.
4. `Run` for game simulation, scheduled in parallel when component access contracts allow it.
5. `Late` for deterministic post-simulation work.
6. `Render` for render submit/main-thread rendering work.
7. `Destroy` once when pipeline is destroyed.

No separate `End` phase in 0.4. `Late` covers post-frame gameplay cleanup until there is a concrete separate responsibility.

## Design Rules

- Parallelism is a scheduler property, not a user-facing system name.
- User code should implement `ISystemRun`, not `ISystemRunParallel`.
- `FixedRun` and `Run` are eligible for parallel scheduling.
- `Begin` and `Render` are sequential/main-thread phases.
- `Late` starts sequential for predictability; parallel late can be added later if a real need appears.
- Unknown component access is treated as conflict and goes through a sequential barrier.
- Main-thread-only systems must have an explicit marker/attribute and must never be scheduled onto worker threads.
- All steady-state phases must be no-GC after warm-up.

## Public API Target

Add Karpik lifecycle interfaces:

- `ISystemInit`
- `ISystemBegin`
- `ISystemFixedRun`
- `ISystemRun`
- `ISystemLateRun`
- `ISystemRenderSubmit`
- `ISystemDestroy`

Preferred injected worlds:

- `DefaultWorld`
- `EventWorld`
- `MetaWorld`

World facade requirements:

- keep `Where<TAspect>(out TAspect aspect)` passthrough for current Dragon aspects;
- add data-first component operations where component data is assigned before lifecycle enable;
- add async lifecycle operations returning `JobHandle`;
- keep direct Dragon world access available only as compatibility/backend escape hatch.

Component lifecycle target:

- engine-owned lifecycle interface, not Dragon `IEcsComponentLifecycle<T>`;
- `EnableAsync` runs after data is assigned;
- `DisableAsync` runs before engine-owned removal/cleanup;
- async lifecycle is explicit: callers receive/await `JobHandle`; the game loop must not block.

## Milestone 1: API Skeleton And Compatibility

Deliverables:

- Add `ISystem*` lifecycle interfaces.
- Add adapter runners so `ISystem*` systems can run inside the existing Dragon `EcsPipeline`.
- Keep existing `IEcs*` systems working.
- Register Karpik runners from `ECS.Core`.
- Document intended phase order in code comments/docs.

Acceptance:

- A test/sample system can implement `ISystemInit`, `ISystemBegin`, `ISystemRun`, `ISystemLateRun`, `ISystemRenderSubmit`, and `ISystemDestroy`.
- Existing modules still start without migration.
- Execution order is deterministic in a simple smoke test.

## Milestone 2: Scheduler Contract For Run

Deliverables:

- Rework current `IEcsRunParallel` prototype into the `ISystemRun` scheduler backend.
- Reuse existing `SystemExecutionNode` aspect reflection as the first access-contract source.
- Build dependency graph at init/rebuild time.
- Treat unresolved access as a sequential conflict.
- Add main-thread/sequential escape hatch.
- Remove avoidable per-frame allocations in scheduling.

Acceptance:

- Systems with disjoint writes can run in parallel.
- Write/read and write/write conflicts are ordered deterministically.
- Systems with unknown access are not run concurrently with unsafe neighbors.
- Scheduler warm path does not allocate per frame.

## Milestone 3: World Facade And Component Lifecycle

Deliverables:

- Extend `World`, `DefaultWorld`, `EventWorld`, and `MetaWorld` as preferred user-facing APIs.
- Add data-first add/set operations.
- Add `JobHandle`-based async enable/disable lifecycle.
- Route engine-owned component creation/removal through the facade where practical.
- Add diagnostics with world, entity id, component type, and lifecycle phase on failure.

Acceptance:

- `AddAsync<T>` assigns the supplied component data before `EnableAsync`.
- `DisableAsync` observes current component data before removal.
- `SpriteRenderer`-style resource setup can use assigned texture path/color/size during enable.
- Direct Dragon `EcsPool.Add` remains compatibility-only and is not used by migrated engine code.

## Milestone 4: Engine Module Migration

Deliverables:

- Convert core ECS runner systems to `ISystem*`.
- Convert representative shared modules: physics fixed/run systems, tween update, ECS maintenance systems.
- Convert representative client modules: graphics begin/render systems, input update, network client update.
- Convert representative server modules: network server update, fixed gameplay systems.
- Keep old `IEcs*` support until samples are migrated.

Acceptance:

- Client starts and renders existing sample.
- Server fixed tick still uses `Application.TICK_DT`.
- Hot reload world snapshot/restore still works.
- No Client project imports Server and no Server project imports Client.

## Milestone 5: Sample Migration And Deprecation

Deliverables:

- Migrate `MyGame` systems to `ISystem*` and Karpik world wrappers where feasible.
- Convert `SpriteRenderer` lifecycle to Karpik component lifecycle.
- Mark Dragon lifecycle interfaces as compatibility/deprecated in docs for user modules.
- Add examples for main-thread `Begin`/`Render`, parallel `Run`, and fixed simulation.

Acceptance:

- Sample gameplay still works after migration.
- User-facing examples do not require `IEcsRun`, `IEcsInit`, or `IEcsDestroy`.
- Aspects may still use Dragon types.

## Milestone 6: Validation And Docs

Deliverables:

- Add tests for phase order.
- Add tests for scheduler conflicts.
- Add tests for lifecycle data-before-enable behavior.
- Add allocation checks for steady-state `Begin`, `FixedRun`, `Run`, `Late`, and `Render`.
- Add docs for lifecycle phases, parallel scheduling rules, and world facade usage.

Acceptance:

- Phase order test proves `Init -> Begin -> FixedRun -> Run -> Late -> Render -> Destroy`.
- Parallel scheduler tests cover disjoint systems, conflicts, unknown access, and main-thread-only systems.
- Lifecycle tests cover async enable, async disable, and exception diagnostics.
- Docs clearly state that Dragon lifecycle APIs are backend/compatibility, not preferred user API.

## Risks

- Aspect reflection is useful as a prototype but incomplete as a long-term contract.
- Parallel scheduling can be slower than sequential execution for tiny systems if graph/job overhead is not controlled.
- Async component lifecycle can make readiness ambiguous if components do not expose loading/ready state.
- Direct raw Dragon pool usage can bypass Karpik lifecycle until migration/analyzers catch it.

## Defaults

- No `End` phase in 0.4.
- `Begin` and `Render` are main-thread sequential.
- `Run` is scheduler-managed and parallel when proven safe.
- `Late` is sequential in the first implementation.
- Unknown access equals conflict.
- Compatibility first, hard removal later.
- Existing Dragon `IEcsRun` execution through `_pipeline.Run()` remains in place for backward compatibility during 0.4 migration.

## 0.4 Exit Gate Before 0.5

- Add a smoke test that proves lifecycle phase order for a minimal system: `Init -> Begin -> FixedUpdate -> Update/Run -> Late -> Render -> Destroy`.
- The smoke test must run before treating lifecycle foundation as complete and moving roadmap focus to 0.5 scheduler/jobs/memory work.
