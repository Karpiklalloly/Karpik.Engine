# 0.4 System Lifecycle Plan

## Summary

Goal: move user modules away from direct DragonECS lifecycle APIs and make KarpikEngine own the system pipeline.

DragonECS remains the internal ECS backend for 0.4. Aspects, `IEcsComponent`, `EcsPool<T>` inside aspects, and existing Dragon internals can stay for now. The public direction is different: user systems implement Karpik `ISystem*` contracts and receive Karpik world facades. Scheduler-managed parallel execution is deferred to 0.5, where it can be designed together with the full multithreaded runtime model.

The frame pipeline:

1. `Init` once when pipeline is built.
2. `Begin` every frame, sequential/main-thread.
3. `FixedUpdate` for fixed simulation.
4. `Update` for game simulation.
5. `LateUpdate` for deterministic post-simulation work.
6. `Render` for render submit/main-thread rendering work.
7. `Destroy` once when pipeline is destroyed.

No separate `End` phase in 0.4. `Late` covers post-frame gameplay cleanup until there is a concrete separate responsibility.

## Design Rules

- Parallelism is a scheduler property, not a user-facing system name. The public scheduler integration is deferred to 0.5.
- User code should implement `ISystemUpdate`, not `IEcsRun` or `IEcsRunParallel`.
- `FixedUpdate` and `Update` are eligible for future scheduler-managed execution.
- `Begin` and `Render` are sequential/main-thread phases.
- `LateUpdate` starts sequential for predictability; parallel late update can be added later if a real need appears.
- The 0.5 scheduler must treat unknown component access as conflict and route it through a sequential barrier.
- The 0.5 scheduler must model thread affinity explicitly; main-thread-only systems must never be scheduled onto worker threads.
- All steady-state phases must be no-GC after warm-up.

## Public API Target

Add Karpik lifecycle interfaces:

- `ISystemInit`
- `ISystemBegin`
- `ISystemFixedUpdate`
- `ISystemUpdate`
- `ISystemLateUpdate`
- `ISystemRender`
- `ISystemDestroy`

Preferred injected worlds:

- `DefaultWorld`
- `EventWorld`
- `MetaWorld`

World facade requirements:

- keep `Where<TAspect>(out TAspect aspect)` passthrough for current Dragon aspects;
- add data-first component operations where component data is assigned before lifecycle enable;
- add async lifecycle operations returning `JobHandle`;
- keep direct Dragon world access available only as compatibility/backend escape hatch.

Component lifecycle target:

- engine-owned lifecycle interface, not Dragon `IEcsComponentLifecycle<T>`;
- `EnableAsync` runs after data is assigned;
- `DisableAsync` runs before engine-owned removal/cleanup;
- async lifecycle is explicit: callers receive/await `JobHandle`; the game loop must not block.

## Milestone 1: API Skeleton And Compatibility

Deliverables:

- Add `ISystem*` lifecycle interfaces.
- Add adapter runners so `ISystem*` systems can run inside the existing Dragon `EcsPipeline`.
- Keep existing `IEcs*` systems working.
- Register Karpik runners from `ECS.Core`.
- Document intended phase order in code comments/docs.

Acceptance:

- A test/sample system can implement `ISystemInit`, `ISystemBegin`, `ISystemFixedUpdate`, `ISystemUpdate`, `ISystemLateUpdate`, `ISystemRender`, and `ISystemDestroy`.
- Existing modules still start without migration.
- Execution order is deterministic in a simple smoke test.

## Milestone 2: Parallel Scheduler Prototype

Deliverables:

- Keep current `IEcsRunParallel` isolated as a prototype; do not integrate it into `ISystemUpdate` during 0.4.
- Reuse existing `SystemExecutionNode` aspect reflection as the prototype access-contract source. Status: implemented.
- Build the prototype dependency graph at init/rebuild time. Status: implemented.
- Treat unresolved access as a sequential conflict in the prototype graph. Status: implemented.
- Remove avoidable per-frame allocations in prototype scheduling. Status: implemented for ECS-side buffers; `Karpik.Jobs` allocation work remains in 0.5.
- Cover graph rules with focused tests. Status: implemented for empty graph, disjoint writes, write/read, read/write, write/write, unknown access, and deterministic registration order.
- Defer `ISystemUpdate` integration, thread-affinity attributes, client loop threading, and full no-GC job scheduling to 0.5.

Acceptance:

- Prototype graph tests prove that disjoint writes have no dependency.
- Prototype graph tests prove that write/read, read/write, and write/write conflicts are ordered deterministically.
- Prototype graph tests prove that unknown access creates a conservative barrier.
- The 0.4 runtime continues to execute `ISystem*` phases sequentially.

## Milestone 3: World Facade And Component Lifecycle

- Add `JobHandle`-based async enable/disable lifecycle. Status: implemented as the single component lifecycle path.
- Route engine-owned component creation/removal through the facade where practical. Status: `SpriteRenderer` is the first migrated reference case.
- Add diagnostics with world, entity id, component type, and lifecycle phase on failure. Status: implemented through `ComponentLifecycleException`.
- Add analyzer coverage for lifecycle bypass. Status: analyzer is connected and checks direct lifecycle component mutation through raw Dragon APIs.

Acceptance:

- `AddEnabledAsync<T>` assigns the supplied component data before `EnableAsync`.
- `DisableAsync` observes current component data before removal.
- `SpriteRenderer` resource setup uses assigned texture path/color/size during enable.
- Direct Dragon `EcsPool.Add` remains compatibility-only and is rejected by analyzer for lifecycle components outside allowed backend/template paths.
- `ComponentsTemplate.OnLoad` remains part of template/json asset loading, not component runtime lifecycle, and is intentionally kept.

## Build And Dependency Hygiene

- Use single-node builds with MSBuild node reuse disabled for agent validation: `dotnet build <project-or-solution> -m:1 -nr:false`.
- `Configurator --validate` verifies Client/Server/Shared project-reference boundaries and returns a non-zero exit code on violations.
- `Directory.Build.targets` runs `Configurator --validate` for launcher/publish builds unless `KarpikSkipValidation=true`.
- `StaticAnalyzer` is centrally applied to `Modules` and `MyGame` projects so system facade violations fail during normal project builds.
- `Newtonsoft.Json` direct package references omit local versions; version `13.0.4` is pinned centrally through `Directory.Build.targets`.

## Risks

- Aspect reflection is useful as a prototype but incomplete as a long-term contract.
- Parallel scheduling can be slower than sequential execution for tiny systems if graph/job overhead is not controlled.
- Async component lifecycle can make readiness ambiguous if components do not expose loading/ready state.
- Direct raw Dragon pool usage can bypass Karpik lifecycle until migration/analyzers catch it.
- Blocking facade shortcuts such as sync `AddEnabled`/`DelEnabled` over async lifecycle are temporary compatibility helpers. Replace them with queued lifecycle operations so callers do not await manually and `Update` does not block on asset loading. `SpriteRenderer` is the reference case: enqueue enable, mark pending/not-ready, await asset load in a lifecycle processor, write the enabled component back, then clear pending state.

## Defaults

- No `End` phase in 0.4.
- `Begin` and `Render` are main-thread sequential.
- `Update` is the public simulation phase and remains sequential in 0.4; future scheduler-managed execution must remain an implementation detail.
- `LateUpdate` is sequential in the first implementation.
- Unknown access equals conflict.
- `IEcsRunParallel` remains an isolated prototype in 0.4. Integration into `ISystemUpdate`, thread-affinity metadata, and the multithreaded client loop belong to 0.5.
- Compatibility first, hard removal later.
- Existing Dragon `IEcsRun` execution through `_pipeline.Run()` remains in place for backward compatibility during 0.4 migration.

## 0.4 Exit Gate Before 0.5

- Add a smoke test that proves lifecycle phase order for a minimal system: `Init -> Begin -> FixedUpdate -> Update -> LateUpdate -> Render -> Destroy`. Status: implemented through `Karpik.Engine.Core.Runner.Tests`.
- The smoke test must run before treating lifecycle foundation as complete and moving roadmap focus to 0.5 scheduler/jobs/memory work.
