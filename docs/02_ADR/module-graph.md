# Deterministic Module Graph

> Status: accepted
> Date: 2026-05-31
> Owners: developer and Codex
> Related ExecPlan: [[../../plans/module-graph-execplan]]

## Context

Module projects previously declared standard `ProjectReference` items while Configurator independently scanned project filenames. The two sources could drift, disabled dependencies were not rejected, generated artifacts could become stale, and side leaks were diagnosed only partially.

This work runs during MSBuild evaluation, Configurator execution, startup, and shutdown. It does not add frame-loop, fixed-tick, ECS-run, render-loop, or network-pump work.

## Decision

Projects under `Modules` and `MyGame` declare source-level project dependencies only with shorthand `KarpikModuleDependency` ids. Logical module ids such as `Physics2D` resolve to their core plugin, implementation ids such as `Physics2D.Aether2D` resolve to a concrete plugin, and infrastructure ids such as `Dragon` resolve by project filename.

Configurator reads raw project XML, derives plugin identity and side from repository conventions, validates the structured profile in `Directory.Build.props`, rejects unknown and ambiguous dependency ids, and generates deterministic `AutoGenerated.targets`, `Generated/KarpikModuleCatalog.props`, and `Generated/ModuleLoader.cs` artifacts. `Directory.Build.targets` imports the catalog and expands dependency ids into evaluated `ProjectReference` items so Rider and the compiler still resolve referenced types after project reload.

Runtime installer lifecycle ordering remains priority-first. Equal priorities use assembly load rank, installer full type name, then registration rank. Installer teardown runs in reverse init order.

## Alternatives Considered

- Keep separate `ProjectReference` and Configurator metadata declarations. Rejected because duplicated dependency state drifts.
- Generate references only after running Configurator. Rejected because IDE type resolution must not depend on a generator run.
- Use compile-DAG order for installer lifecycle. Rejected because existing installer priorities intentionally differ from compile dependencies.

## Consequences

- New modules must be added explicitly to the structured profile.
- Adding or moving a project requires regenerating the tracked catalog.
- Invalid naming, side leaks, missing dependencies, disabled required dependencies, cycles, and stale artifacts fail before runtime.
- Configurator allocates model objects and loader startup creates descriptors; hot paths remain unchanged.
- Rider must reload projects after dependency declaration edits so evaluated `ProjectReference` items refresh.
