# План рефакторинга Graphics модуля

## Цель
Переход от "сырой" реализации к высокопроизводительному асинхронному рендерингу с поддержкой батчинга и Zero Allocation в Hot Path.

---

## Этап 1: Рефакторинг Command Buffer (API)
Переход от "толстой" структуры `DrawCommand` к эффективному generic-хранилищу.

- [ ] **Generic API:** Изменить `ICommandBuffer`, добавив `void Add<T>(in T cmd) where T : struct`.
- [ ] **Typed Storage:** Переписать `ThreadBuffer`. Вместо одного массива `DrawCommand[]` использовать типизированные хранилища (например, `ArrayPool` или просто отдельные массивы для каждого типа команд: `Rect`, `Texture`, `Text`).
- [ ] **Thread-Local Access:** Оптимизировать `GraphicsContext.Buffer`, минимизировав блокировки (использовать `ThreadStatic` для кэширования буфера текущего потока).
- [ ] **Удаление Fat Struct:** Выпилить общую структуру `DrawCommand` в пользу легковесных `DrawRectCmd`, `DrawTextureCmd` и т.д.

## Этап 2: Оптимизация рендеринга (Batching)
Главная цель — уменьшить количество Draw Calls и исключить `UpdateBuffer` на каждый объект.

- [ ] **Dynamic Batching:** Реализовать сборщик батчей в `MergeThread`.
    - Собирать данные всех `DrawRectCmd` в один большой `VertexBuffer` (напр. до 10,000 вершин за раз).
    - Использовать один `DrawCall` для отрисовки всех примитивов с одинаковым шейдером/текстурой.
- [ ] **Instance Buffer:** (Опционально) Рассмотреть использование Instancing для стандартных фигур.
- [ ] **Resource Sets:** Оптимизировать переключение текстур (сортировка команд по материалу перед мерджем).

## Этап 3: Жизненный цикл и Асинхронность
Упорядочивание процесса подготовки и отправки кадра.

- [ ] **Устранение дублирования:** Удалить логику мерджа из `VeldridGraphicsContext.Submit`. Вся работа по сливанию буферов должна быть только в `MergeThread`.
- [ ] **Double Buffering:** Реализовать использование двух `CommandList`. Пока один отправляется в GPU, второй может заполняться мердж-потоком для следующего кадра (или следующего этапа).
- [ ] **Async Merge:** Убедиться, что `MergeThread.BeginMerge()` действительно не блокирует Main Thread, а `WaitForCompletion()` вызывается только перед `Submit`.

## Этап 4: Интеграция и Тесты
- [ ] **ECS Systems:** Обновить `GraphicsCoreBeginSystem` (очистка) и создать систему завершения кадра.
- [ ] **Unit-тесты:**
    - Тест на Zero Allocation при добавлении 1,000,000 команд.
    - Тест на корректность сборки батчей (правильное количество вершин).
- [ ] **Stress Test:** Создать бенчмарк-сцену с 10,000+ динамическими прямоугольниками.

## Архитектурные изменения в коде
```csharp
// Было (медленно, много памяти)
buffer.Add(new DrawCommand { Type = Rect, ... });

// Станет (быстро, кэш-френдли)
buffer.Add(new DrawRectCmd { ... });
```

## Текущие риски
1. **ThreadStatic Cleanup:** Нужно аккуратно очищать буферы всех потоков в начале кадра, чтобы не было утечек памяти или "мусора" из прошлого кадра.
2. **Batch Limits:** При переполнении батч-буфера (например, более 65к вершин) нужно принудительно делать Submit и начинать новый батч.
