# v0.3

## Главные изменения
* **Ассеты:** Добавлена система для загрузки и выгрузки ресурсов.
* **Выполнение в главном потоке:** Добавлен MainThreadSheduler, чтобы выполнять логику в главном потоке, например, для работы с графикой.
* **Архитектура проекта:** Проект разделен на модули и элементы игры. Принципиального различия нет, кроме того, что MyGame.**Side**.Main предполагает использование всех доступных модулей и подпроектов MyGame. В то время как Модули ссылаются только на необходимые им другие модули.
* **Karpik.Jobs:** Внедрены по всему проекту.
* **Горячая перезагрузка:** Добавлена поддержка горячей перезагрузки. Полная. Без ограничений стандартного .NET.

## Ассеты
* Добавлен модуль AssetManagement. Используется для единой загрузки и выгрузки ассетов (текстур, конфигов).
* Использование можно посмотреть в SpriteRenderer и DemoModuleClient.

## Выполнение в главном потоке
* Добавлен класс MainThreadScheduler. Нужен для выполнения действий в главном потоке.

## Архитектура проекта
* Проект разделен на Модули и Игру. Каждый из них делится на Клиентские, Серверные и Общие.
* Модули предполагают обобщенную логику, не зависящую от конкретной игры. Например, AssetManagement, который используется в каждой игре.
* Модули могут реагировать на разные события, подробнее в IModule.cs проекта Karpik.Engine.Core.Runner. Обязательными являются ModuleAttribute и IModule.
* Для Игровых проектов поведение аналогично.
* Если вы добавляете новый Модуль или Игровой проект, то запустите команду `dotnet run --project Configurator/Configurator.csproj -- --generate`. Она автоматически сгенерирует файлы, основываясь на всех проектах.

## Горячая перезагрузка
* О да, на это ушло больше всего времени.
* На данный момент для Горячей перезагрузки используется отдельный процесс, в который загружаются все библиотеки.
* Для корректной работы в Debug режиме включите в настройках Вашей IDE автоматическое подключение Дебаггера к дочерним процессам.
* Как это работает: Вы запускаете игру, у Вас неправильно работает игровая логика, Вы изменяете логику, Вы вызываете Сборку проекта, **проверяете обновились ли файлы в каталоге modules (обычно нужно вызвать Сборку 2 раза)**, Нажимаете на `Hot Reload` в дебаг панели.
* По умолчанию сохраняются все сущности в каждом из ECS миров, поэтому рекомендуется сохранять все данные именно в ECS мирах. Также, можно настроить и свою вариацию сохранения.


# v0.2

## Главные изменения
* **UI:** Добавлена система UI, вдохновленная html+css. Подробнее в вики.
* **Внедрение Dependency Injection:** Большинство статических классов удалено и заменено на внедрение зависимостей. Реализована система `AutoInject` для систем.
* **Karpik.Jobs:** Импортирована и интегрирована система джобов.
* **Tween:** Добавлены твины (GTweens).


## Архитектура и ECS
* Добавлены `EcsCommandBuffer`, `BaseSystem` и `IEcsRunParallel`.
* Переименованы проекты и добавлен файл `Game.targets`.

## Сеть и Моддинг
* **Сеть:**
    * Исправлена работа RPC.
    * Добавлен компонент `LocalPlayer` для идентификации локального игрока.
    * Реализован автоматический выбор порта для клиента.
* **Моддинг:**
    * Добавлено разделение на клиентские и серверные подпапки для модов.
    * Добавлены ссылки на папки `Mods` и `Content` в Debug-сборке.
    * Исправлен расчет delta time (dt) на сервере.
